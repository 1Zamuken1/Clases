El Poder de la Abstracción

Antes de dominar SOLID o Arquitectura, debes entender su átomo fundamental: La Interfaz. Definir QUÉ se hace, no CÓMO se hace.

¿Qué es un Contrato?

Imagina un enchufe de pared. No te importa si la electricidad viene de energía solar, eólica o nuclear. Solo te importa que cumpla el contrato: "Si me enchufas (input), te doy 220V (output)".

"Programar contra una interfaz, no contra una implementación."

— Gang of Four

Evolución en Java

  • Interface Clásica:

    Define métodos públicos abstractos. El estándar de oro.

  • FunctionalInterface:

    Un solo método abstracto. Base de Lambdas y Streams.

  • Sealed Interface (Java 17+):

    Control total sobre quién puede implementarla.

Modern Java: Sealed Interfaces

Java 21 Feature

¿Por qué usar Sealed?

A veces no quieres que "cualquiera" implemente tu interfaz. Quieres un conjunto finito y controlado de opciones. Esto permite al compilador saber exactamente qué casos existen, haciendo innecesario el bloque default en un switch.

Seguridad de tipos en tiempo de compilación
Exhaustividad en Pattern Matching
Modelado preciso de dominio 
// ✅ CONTRATO CERRADO (Java 21 Sealed Interface)
public sealed interface PaymentMethod 
    permits Paypal, Bitcoin, ApplePay {}

// CLASES SEPARADAS (Cada una encapsula su lógica)
public final class Paypal implements PaymentMethod { 
    public void sendFunds() { ... } 
}
public final class Bitcoin implements PaymentMethod { ... }

// PROCESADOR (No cambia lógicamente, solo la estructura switch)
public class PaymentService {
    public void process(PaymentMethod pm) {
        // Pattern Matching switch (Exhaustivo - No requiere default)
        switch (pm) {
            case Paypal p -> p.sendFunds();
            case Bitcoin b -> b.transferHash();
            case ApplePay a -> a.authorize();
        }
    }
}
