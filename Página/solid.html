Principios SOLID

Reglas Universales para software mantenible.

"Una clase debe tener una, y solo una, razón para cambiar."

Profundización

No mezcles lógica de negocio con infraestructura. Si cambia la base de datos, tu lógica de negocio no debería tocarse.

Analogía: El Chef

Un chef cocina. No repara las tuberías del gas. Si hace ambas cosas, ¡se le quema la cena!

Tip para el Aprendiz

Si describes tu clase con una "Y" ("Valida Y guarda"), está haciendo demasiado.

// ❌ PAQUETE: com.monolith.service
import java.sql.*;

public class OrderService {
    public void processOrder(int id) {
        // 1. NEGOCIO: Cálculos
        double total = 100.0 * 1.21;

        // 2. INFRA: Guardar en BD (Acoplado a JDBC)
        try (Connection c = DriverManager.getConnection("jdbc:mysql...")) {
            c.createStatement().execute("UPDATE orders...");
        } catch (SQLException e) { e.printStackTrace(); }

        // 3. INFRA: Enviar Email (Acoplado a implementación)
        System.out.println("Enviando correo...");
    }
}
// ✅ PAQUETE: com.clean.usecase
// Solo orquesta. Delega los detalles feos a interfaces.

public class ProcessOrderUseCase {
    private final OrderRepository repo;
    private final Notifier notifier;
    
    public ProcessOrderUseCase(OrderRepository r, Notifier n) {
        this.repo = r;
        this.notifier = n;
    }

    public void execute(Order order) {
        order.calculateTotal();      // Responsabilidad del Dominio
        repo.save(order);            // Responsabilidad de Infra
        notifier.sendReceipt(order); // Responsabilidad de Notificación
    }
}

"Abierto a extensión, cerrado a modificación."

Profundización

Debes poder añadir nueva funcionalidad (nuevo método de pago) creando clases nuevas, NO editando un `switch/if` gigante existente.

Analogía: La Consola

Para jugar un juego nuevo, compras un cartucho (Extensión). No necesitas desarmar y soldar la consola (Modificación).

// ❌ PAQUETE: com.bad.notification
public class NotificationSender {
    public void send(String type, String message) {
        // Si pidiéramos 'WhatsApp', ¡toca abrir y modificar esta clase!
        if (type.equals("EMAIL")) {
            // logica email...
        } else if (type.equals("SMS")) {
            // logica sms...
        }
    }
}
// ✅ PAQUETE: com.good.notification
// Definimos el contrato. Nuevos canales = Nuevas clases (Extensión).

public interface NotificationChannel {
    void send(String msg);
}

public class EmailChannel implements NotificationChannel {
    public void send(String msg) { /* SMTP logic */ }
}

public class WhatsAppChannel implements NotificationChannel {
    public void send(String msg) { /* API logic */ }
}

"Las subclases deben ser sustituibles por sus padres sin romper nada."

Profundización

Herencia no es solo reutilizar código. Es una relación "ES-UN". Si sobrescribes un método para lanzar `UnsupportedOperationException`, violas LSP.

Analogía: El Pato

Si parece pato y hace cuack, pero necesita baterías... NO es un pato, es un juguete. No puedes tratarlo igual.

// ❌ PAQUETE: com.bad.payments
class TarjetaCredito {
    void pagarCuotas() { /* ... */ }
}

// El hijo NO cumple el contrato del padre. Sorpresa desagradable.
class TarjetaDebito extends TarjetaCredito {
    @Override
    void pagarCuotas() {
        throw new UnsupportedOperationException("¡Débito no tiene cuotas!");
    }
}
// ✅ PAQUETE: com.good.payments
// Segregamos por capacidades. No forzamos herencia incorrecta.

interface InstrumentoPago { void pagar(); }
interface ConCuotas { void pagarCuotas(); }

class TarjetaDebito implements InstrumentoPago { /* Solo pagar */ }
class TarjetaCredito implements InstrumentoPago, ConCuotas { /* Ambas */ }

"Los clientes no deben depender de interfaces que no usan."

Profundización

Mejor tener 3 interfaces pequeñas y específicas (Role Interfaces) que 1 interfaz gigante "Basurero" con 50 métodos.

Analogía: La Navaja Suiza

Si solo necesitas cortar pan, no quieres cargar con una navaja que también tiene sacacorchos, lima y tijeras.

// ❌ PAQUETE: com.bloated.interfaces
// Este 'Dispositivo' obliga a implementar cosas que no tienes.
interface SmartDevice {
    void print();
    void fax(); // ¿Quién usa fax hoy día?
    void scan();
}

class BasicPrinter implements SmartDevice {
    public void print() { /* ok */ }
    public void fax() { throw new Error("No tengo fax"); } // Feo
}
// ✅ PAQUETE: com.clean.interfaces
// Interfaces pequeñas y específicas.
interface Printer { void print(); }
interface Scanner { void scan(); }

class BasicPrinter implements Printer {
    public void print() { /* Imprime feliz */ }
}

class Photocopier implements Printer, Scanner { /* Hace ambos */ }

"Depende de abstracciones, no de detalles."

Profundización

El módulo de alto nivel (Negocio) no debe importar al de bajo nivel (MySQL). Ambos deben importar una Interfaz (Puerto).

Analogía: La Lámpara

No sueldas la lámpara directamente al cableado de la pared. Usas un enchufe estándar. Así puedes cambiar la lámpara sin romper la pared.

Tip Hexagonal

Esta es la base absoluta de la arquitectura Hexagonal (Puertos y Adaptadores).

// ❌ PAQUETE: com.tightly.coupled
import com.infra.disk.FileLogger;

public class AuditService {
    // Dependencia directa de una clase concreta (Bajo Nivel)
    // Si queremos cambiar a DatabaseLogger, ¡hay que romper esta clase!
    private FileLogger logger = new FileLogger();
    
    public void audit() {
        logger.writeToFile("Acción...");
    }
}
// ✅ PAQUETE: com.clean.dip
import com.domain.ports.Logger;

public class AuditService {
    private final Logger logger;

    // INYECCIÓN: Alguien más decide qué logger darme.
    public AuditService(Logger logger) {
        this.logger = logger;
    }
    
    public void audit() {
        logger.log("Acción..."); // No sé donde se guarda, ni me importa.
    }
}
